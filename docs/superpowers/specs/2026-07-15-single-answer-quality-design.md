# Exo 單一完整回答與內容品質設計

日期：2026-07-15
狀態：Evan 已核准對話設計，待規格文件最終審閱

## 背景與問題

目前 `work`、`daily` 的 prompt 強制輸出 2–3 條編號建議，Worker 又把短／中／長限制成每條 10／20／40 個中文字。實機測試因此出現三個問題：

1. 同一個問題被拆成三條薄弱回答，而不是一段可直接說出口的完整論述。
2. 每條內容太短，容易重述問題，無法表現專業判斷與推理。
3. 模型為了填滿條目，捏造「我們已驗證」與精確成效數字。

本設計將所有模式改成單一完整答案，提升內容密度，並明確限制未有來源的個人／公司實績與數據。

## 目標

- `work`、`daily`、`custom` 都只輸出一個可直接照念的完整答案。
- 第一部分直接表態，後續以原因、機制、影響、演進或例子展開。
- 工作模式能自然使用相關專業術語，呈現專業性而不堆砌英文。
- 可以運用通用專業知識，但不能把假設或常識冒充成 Evan／公司做過的事。
- 保留現有 Anthropic 純文字串流、手機即時顯示及眼鏡 300ms 節流。
- 不改變 Worker API 的外部回傳型別，降低改動風險。

## 非目標

- 本次不新增眼鏡手動翻頁或改變手勢語意。
- 本次不修改 `src/even.ts` 的 `enqueue()` 與渲染序列化機制。
- 本次不改模型選擇、知識庫掛載、場景說明或收音流程。
- 本次不以後處理器重寫模型內容；品質主要由明確輸出契約與 prompt 約束。

## 方案選擇

採用「單一答案契約」：保留 `{ suggestions: string[] }` API，但所有模式的成功回覆固定只有一個元素。沒有採用只改 prompt 的方案，因為現有 Plugin 在串流完成後仍會解析編號並補回 `1.`；也不改成 `{ translation, answer }` 新 schema，避免擴大 Worker、transport 與 fallback 的改動面。

## 輸出契約

### 中文模式

- 回覆是一段完整答案，不使用 `1. 2. 3.` 編號。
- 開頭 1–2 句直接提出判斷或立場，不以重述問題湊字數。
- 後續依題目選用原因、運作機制、影響、演進或例子，使內容形成一條連續論述。
- 題目適合時，結尾補一個關鍵風險、限制或落地條件；不強制每題都做正反並列。

### 英文模式

- 第一行固定為 `譯：<對方那句話的中文翻譯>`。
- 下一段只給一個完整英文回答，使用 CEFR B1 以內的句型與詞彙。
- 翻譯與英文回答共同存在 `suggestions[0]`，不得拆成多個選項。

### 自訂模式

- 自訂 prompt 決定角色、立場、語氣及內容方向。
- Worker 在自訂 prompt 後仍附加共同契約：單一完整答案、長度、語言格式及事實邊界。
- 自訂模式不再擁有覆寫「單一答案」與事實安全規則的權限。

## 內容與事實政策

所有模式共用以下規則：

1. 直接回答核心問題，不只是換句話說。
2. 題目相關時自然加入 2–4 個專業英文術語，例如 `READ ONLY`、`TAKE ACTION`、`LLM`、`MCP`、`API` 或 `human-in-the-loop`；不可為了顯得專業而堆砌。
3. 可以使用通用專業知識、原理與公開概念。
4. 假設案例必須用「例如」等字樣明確標示，不得暗示是 Evan 或其公司已完成的案例。
5. 除非逐字稿、場景說明或已掛載知識庫明確提供，不得聲稱「我／我們做過、驗證過」或提供具體成效數字。
6. 資料不足時改談機制、原則、條件或風險，不用虛構內容填滿篇幅。

模式差異只影響語氣：

- `work`：專業、有結構、有明確觀點；避免無意義的猶疑語。
- `daily`：自然、口語、像真人交談，但仍需有實質內容。
- `custom`：遵循自訂角色與語氣，再套用共同契約。

## 長度規則

長度限制針對「單一答案全文」，不再是每一條建議：

| 設定 | 中文 | 英文（不含翻譯行） |
|---|---:|---:|
| 短 | 80–120 個中文字 | 30–45 words |
| 中（預設） | 180–240 個中文字 | 70–100 words |
| 長 | 320–420 個中文字 | 130–170 words |

字數是目標範圍，不要求模型為湊到下限而重複或虛構；內容完整與事實正確優先。

## 資料流與相容性

1. Plugin 繼續傳送 `mode`、`transcript`、`customPrompt`、`sceneNote`、`length`、`lang` 及已勾選 KB。
2. Worker 依模式建立基礎 prompt，再依既有順序加入場景、KB、延伸內容、單一答案契約、長度、語言與去重規則。
3. Anthropic 串流維持 `text/plain`；Plugin 在 `onDelta` 中把累積全文當成唯一答案即時顯示。
4. 串流完成後，transport 不再把全文解析成多個編號項目，而是回傳 `[完整全文]`。
5. 非串流 Anthropic／OpenAI fallback 同樣回傳 `[完整全文]`。
6. `main.ts` 收到單一元素後直接顯示原文，不補 `1.`；recent-suggestions 去重緩衝仍以該完整答案為一筆。
7. API 型別維持 `suggestions: string[]`，避免影響現有呼叫邊界。

若模型違反 prompt 自行產生編號，transport 仍保留完整文字為一個元素，不把它拆成多個替代答案。模型格式違規由 prompt 測試與實機驗收追蹤，不用脆弱的文字重寫器猜測語意。

## 顯示行為

- 手機 DOM 每次收到 delta 都顯示完整累積文字，完成後保留全文。
- 眼鏡更新仍走 `even.ts` 的 `enqueue()`，並維持至少 300ms 節流。
- 現行 512-byte 保守尾端視窗不變。中／長回答會在串流時自動向後捲動，完成後眼鏡保留尾端；手機保留完整全文。
- 本次不啟用 swipe 翻頁。若實機使用顯示無法跟上朗讀，再另立分頁設計，不與內容品質改動綁在一起。

## 錯誤與退化處理

- 空白模型回覆一律視為 LLM error，不建立空的 `suggestions[0]`，也不顯示空答案。
- Anthropic 串流不可用時，沿用非串流 JSON fallback；兩條路徑都遵守單一答案陣列。
- 舊版 Worker 若仍回多元素陣列，Plugin 不再為元素額外補編號；正式驗收以 Plugin 與 Worker 同版部署為準。
- prompt 不保證模型永不犯錯，因此實機驗收必須包含「無來源數字／實績」檢查。

## 測試設計

### 單元與整合測試

- 模式測試：`work`、`daily`、`custom` 的最終 prompt 都包含單一答案與不編號規則。
- prompt 組裝測試：確認場景、KB、自訂 prompt、長度、語言與事實政策的順序和內容。
- 長度測試：中文與英文的短／中／長範圍正確，英文翻譯行不計入。
- 事實政策測試：prompt 明確禁止虛構個人／公司實績與具體數據，並要求假設案例標示「例如」。
- transport 測試：串流單段、多行英文格式及非串流回覆都只產生一個元素。
- main DOM／bridge 測試：完成後不顯示 `1.`，手機與眼鏡接收同一份完整答案。
- mock 測試：fixture 改為每模式一個完整回答，維持本機與 simulator 可重現。

每次實作改動後執行 `npm test`；完成前再執行 `npm run build` 與適用的 Worker／模擬器回歸。

### G2 實機驗收

使用工作模式、中文、中等長度，重問：

> 現在人們期待 AI Agent 能夠在企業裡實際執行任務、串工具、做到很多事情，你怎麼看這個轉變？

合格條件：

- 只有一個完整答案，無 `1. 2. 3.`。
- 約 180–240 個中文字。
- 開頭直接表態，後續解釋從問答型 AI 到 agentic workflow 的技術條件與演進。
- 自然使用 2–4 個相關術語，不堆砌。
- 不聲稱 Evan／公司已驗證某流程，不捏造百分比或成效。
- 適合時補一個落地條件，例如權限治理、可觀測性或 `human-in-the-loop`。
- 手機顯示完整全文；眼鏡串流穩定、BLE 不斷線。
